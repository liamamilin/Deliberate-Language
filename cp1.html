2  Framing the Problem

2.1 The Problem Before the Problem

Deliberate thinking does not begin with solving.

It begins with framing.

Before a person compares alternatives, evaluates risks, forms judgment, or chooses a strategy, something more basic has already happened: a situation has been turned into a problem.

This transformation is not automatic.

Reality presents events, pressures, symptoms, conflicts, failures, desires, and disturbances. But none of these becomes a “problem” until language gives it a form.

A problem is not merely something unpleasant.
It is not merely something difficult.
It is not merely something that demands attention.

A problem is a structured object of thought.

To frame a problem is to say:

  • what the issue is

  • where its boundary lies

  • what caused or shaped it

  • who or what is involved

  • what matters about it

  • what kind of response it calls for

This means that problem framing is not a neutral introduction to thinking. It is already an act of judgment.

The way a problem is stated determines what can be seen, what can be ignored, what can be blamed, what can be changed, and what kinds of solutions appear reasonable.

A poorly framed problem can make excellent reasoning useless.

A well-framed problem does not solve the issue by itself, but it gives thought a stable object to work on.

2.2 A Problem Is a Linguistic Construction

In ordinary speech, people often treat problems as if they simply exist.

They say:

“The problem is money.”
“The problem is time.”
“The problem is motivation.”
“The problem is communication.”
“The problem is the market.”
“The problem is the system.”

These statements feel clear because they name something.

But naming is not yet framing.

To name a problem is to point toward an area of concern.
To frame a problem is to define its structure.

For example:

“The problem is time.”

This sentence is too compressed. It may refer to many different structures:

  • there is not enough time

  • time is being used inefficiently

  • priorities are unclear

  • tasks are overcommitted

  • deadlines are unrealistic

  • sequencing is wrong

  • attention is fragmented

  • the wrong work is being done at the wrong stage

Each formulation opens a different path of thought.

If the problem is “not enough time,” the solution may be to reduce scope.
If the problem is “poor sequencing,” the solution may be to reorder work.
If the problem is “unclear priorities,” the solution may be to define criteria.
If the problem is “fragmented attention,” the solution may be to redesign the working environment.

The same situation can generate different problems depending on how it is framed.

This is the first principle of deliberate language:

A problem is not only found in reality.
It is constructed through language.

2.3 What a Frame Does

A frame is a structure that makes a situation thinkable.

It selects, organizes, and emphasizes.

When language frames a problem, it performs several operations at once.

It defines the object of concern.
It draws boundaries around the issue.
It identifies relevant causes.
It assigns agency or responsibility.
It suggests what kind of action is appropriate.
It implies what would count as improvement.

For example:

“Students are lazy.”

This frame makes student character the central object. It directs attention toward discipline, attitude, and motivation.

But consider another formulation:

“The learning environment does not produce enough feedback.”

Now the frame changes. The object is no longer student character, but system design. Attention moves toward incentives, feedback loops, curriculum structure, and learning conditions.

Another formulation:

“Students do not understand why the material matters.”

This frame moves the problem toward meaning, relevance, and motivation.

Another:

“The course assumes background knowledge that students do not yet have.”

Now the issue becomes prerequisite structure.

Each sentence creates a different problem.

The facts may overlap, but the path of thought changes.

This is why framing is powerful: it does not merely describe a problem after it exists. It helps create the problem that will be reasoned about.

2.4 The Hidden Structure of a Problem Statement

A problem statement usually contains more structure than it appears to contain.

Even a short sentence may imply an entire architecture of thought.

Consider:

“The team is moving too slowly.”

This sentence contains several hidden assumptions:

  • there is a desirable speed

  • the current speed is below that standard

  • the team is the relevant unit of analysis

  • slowness is the key issue

  • speed is the dimension being evaluated

  • improvement means acceleration

But the same situation could be framed differently:

“The team lacks decision clarity.”

Now the issue is not speed itself, but decision structure.

Or:

“The project has too many unresolved dependencies.”

Now the issue is not the team’s performance, but project architecture.

Or:

“The organization rewards caution more than delivery.”

Now the issue becomes incentive design.

Each statement does more than describe. It configures the field of possible explanation and action.

A problem frame usually contains the following elements:

Element Question Function
Object What is the issue about? Defines the center of attention
Boundary Where does the problem begin and end? Prevents uncontrolled expansion
Condition What is the current state? Describes the situation
Standard Compared to what is this a problem? Establishes why it matters
Cause What produces or sustains it? Guides explanation
Agent Who or what can act? Defines responsibility and leverage
Consequence What happens if it continues? Establishes stakes
Direction What kind of change is needed? Opens the path toward solution

A strong problem frame makes these elements explicit.

A weak problem frame leaves them hidden.

2.5 Symptoms, Problems, and Causes

One of the most common failures in deliberate thinking is confusing symptoms with problems.

A symptom is what appears.
A problem is the structured condition that requires attention.
A cause is what produces or maintains that condition.

For example:

“Sales are declining.”

This is a symptom.

It may point to different problems:

  • the product no longer fits customer needs

  • the sales channel is weakening

  • competitors have improved

  • pricing is misaligned

  • the brand has lost trust

  • the market itself is shrinking

  • the sales team lacks capability

  • customer acquisition costs have become too high

Each problem has different causes and different responses.

If a symptom is mistaken for the problem, action becomes shallow.

The organization may say:

“Sales are declining, so we need more marketing.”

But if the real issue is product-market mismatch, more marketing only accelerates exposure to a weak offer.

Or it may say:

“Sales are declining, so the sales team must work harder.”

But if the market has shifted, effort does not solve structural misalignment.

A symptom demands observation.
A problem demands framing.
A cause demands investigation.

Deliberate language must keep these layers separate.

2.6 Problem Framing as Attention Design

To frame a problem is to design attention.

Every formulation tells the mind where to look.

A frame can direct attention toward:

  • individuals

  • systems

  • incentives

  • processes

  • resources

  • timing

  • constraints

  • values

  • communication

  • coordination

  • external conditions

For example:

“He failed.”

This directs attention toward the individual.

“The process failed.”

This directs attention toward workflow.

“The incentive structure produced the wrong behavior.”

This directs attention toward systemic forces.

“The environment made the correct behavior difficult.”

This directs attention toward context.

None of these frames is automatically correct. Each may reveal something or hide something.

The function of deliberate language is not to choose frames casually, but to test them.

A mature thinker asks:

  • What does this frame make visible?

  • What does it hide?

  • What does it assume?

  • What action does it encourage?

  • What alternatives does it exclude?

A frame is useful only when it helps thought contact the real structure of the situation.

2.7 The Boundary of the Problem

A problem without boundaries becomes unmanageable.

If the boundary is too narrow, the real issue is missed.
If the boundary is too wide, the problem becomes impossible to act on.

For example:

“I cannot learn English well.”

This frame is too broad.

It may include pronunciation, vocabulary, grammar, listening, reading, speaking, writing, confidence, learning method, input volume, feedback quality, or daily environment.

A better frame might be:

“My spoken English is weak because I cannot quickly organize simple explanations in real time.”

This is narrower and more actionable.

It defines:

  • the domain: spoken English

  • the specific function: real-time explanation

  • the difficulty: organizing simple explanations quickly

  • the likely direction: practice with structured expression

A bounded problem can be worked on.

But the boundary must not become artificially narrow.

If someone says:

“I need to memorize more words.”

This may be too narrow if the deeper problem is not vocabulary quantity, but the inability to organize meaning into scenes, events, states, and relations.

Good framing therefore requires boundary adjustment.

The problem must be large enough to include the real cause, but small enough to support action.

2.8 The Standard Behind the Problem

Something becomes a problem only in relation to a standard.

Without a standard, there is only description.

Consider:

“The project took three months.”

Is this a problem?

It depends.

If the expected duration was one month, it may indicate delay.
If the expected duration was six months, it may indicate efficiency.
If the project was exploratory, three months may be normal.
If the project was urgent, it may be unacceptable.

A problem statement always implies a gap:

Current State ≠ Desired State

The desired state may be explicit or hidden.

For example:

“The response time is slow.”

Hidden standard:

“The response time should be faster than it currently is.”

But how fast?

  • under one second?

  • under three seconds?

  • fast enough for casual use?

  • fast enough for real-time interaction?

  • fast enough to compete with other products?

The standard matters because it determines the seriousness of the problem and the direction of solution.

A vague standard produces vague judgment.

Deliberate language requires the speaker to ask:

  • What is the expected state?

  • Who defines that expectation?

  • Is the standard realistic?

  • Is the standard measurable?

  • Is the standard appropriate for this context?

Without a clear standard, the problem cannot be evaluated.

2.9 The Difference Between Problem and Discomfort

Not every discomfort is a problem in the deliberate sense.

Discomfort is felt.
A problem is formulated.

Discomfort may be the starting signal, but it is not yet an object of judgment.

For example:

“I feel stuck.”

This is emotionally real, but structurally incomplete.

It may become different problem frames:

  • “I do not know which goal to prioritize.”

  • “I have too many possible paths and no selection criteria.”

  • “I lack a concrete next action.”

  • “I am avoiding a decision because the consequences are uncertain.”

  • “My current environment does not support the identity I am trying to build.”

Each formulation converts discomfort into a workable structure.

This conversion is central to deliberate language.

Language takes a vague pressure and turns it into a defined object of thought.

Until this happens, the mind can only circle the feeling.

Once it happens, judgment can begin.

2.10 Types of Problem Frames

Different problems require different forms of framing.

A problem may be framed as a gap, a conflict, a constraint, a risk, a failure, a trade-off, or an opportunity.

Each form creates a different reasoning path.

Frame Type Basic Form Example Typical Direction
Gap Current state differs from desired state “Our output quality is below the required standard.” Improve capability or process
Conflict Two demands compete “We need speed, but also reliability.” Trade-off and prioritization
Constraint A limit blocks action “We cannot expand because capital is limited.” Resource design or constraint removal
Risk Future harm is possible “If dependency X fails, the system may collapse.” Prevention or contingency
Failure Something did not work as expected “The launch failed to retain users.” Diagnosis and correction
Opportunity A possible gain is visible “There is an unmet need in this market.” Exploration and strategy
Ambiguity Meaning or direction is unclear “Different teams define success differently.” Clarification and alignment
Dependency One element relies on another “The plan depends on a partner we do not control.” Dependency management

The same situation can often be framed in more than one way.

For example, a startup with limited money may say:

“We do not have enough funding.”

This is a constraint frame.

But it may also say:

“We need to find a path that produces learning before capital runs out.”

This is a strategic frame.

The second formulation does not deny the constraint, but it organizes thought around action and sequencing.

A good frame does not merely state difficulty. It creates a path for reasoning.

2.11 How Frames Produce Action Paths

Every problem frame implies an action path.

This is one of the most important principles of this chapter.

A formulation is never passive. It quietly suggests what should be done next.

Consider these different frames:

Problem Frame Implied Action
“The employee is underperforming.” Correct, train, replace, or manage the employee
“The role is poorly defined.” Redesign responsibilities and expectations
“The team lacks feedback loops.” Build review, measurement, and correction mechanisms
“The workload exceeds available capacity.” Reduce scope, add resources, or change priorities
“The system rewards the wrong behavior.” Redesign incentives
“The goal is unclear.” Clarify desired outcomes and criteria

This is why problem framing has practical consequences.

It determines not only how people think, but what they attempt to change.

A frame aimed at personality produces moral judgment.
A frame aimed at process produces operational redesign.
A frame aimed at incentives produces structural intervention.
A frame aimed at goals produces clarification.
A frame aimed at constraints produces negotiation or redesign.

When people disagree about solutions, they often disagree earlier than they realize.

They are not only choosing different actions.
They are operating inside different problem frames.

2.12 Reframing

Reframing is the act of changing the structure of the problem.

It does not mean pretending the issue is positive.
It does not mean using softer language.
It does not mean escaping reality.

Reframing means altering the object, boundary, cause, standard, or action direction of a problem so that thought can contact the situation more accurately.

A useful reframe may move:

  • from symptom to cause

  • from blame to structure

  • from event to pattern

  • from individual to system

  • from impossibility to constraint

  • from vague concern to concrete gap

  • from isolated action to strategic sequence

For example:

“I am bad at writing.”

This is a self-labeling frame.

A reframe:

“I lack a stable process for turning scattered ideas into structured paragraphs.”

Now the problem becomes operational.

Another example:

“The market does not want our product.”

This may be too final.

A reframe:

“We have not yet identified which user segment has a strong enough pain to adopt this product.”

Now inquiry becomes possible.

Another:

“The plan failed.”

A reframe:

“The plan depended on an assumption that turned out to be false.”

Now revision becomes possible.

Reframing is not cosmetic optimism.
It is structural correction.

2.13 Bad Frames and Their Costs

A bad problem frame can cause damage even when everyone involved is intelligent.

Several failure modes are especially common.

2.13.1 The Label Frame

This frame turns a complex situation into a fixed label.

Examples:

“He is lazy.”
“The team is incompetent.”
“The product is bad.”
“The market is impossible.”

Labels stop inquiry.

They may contain partial truth, but they usually compress too much. Once a label is accepted, people stop asking what structure produced the observed behavior.

2.13.2 The Symptom Frame

This frame treats the visible signal as the problem itself.

Examples:

“Traffic is down.”
“Revenue is down.”
“Users are leaving.”
“Students are quiet.”

Symptoms matter, but they are not enough.

A symptom frame creates reactive action before diagnosis is complete.

2.13.3 The Blame Frame

This frame organizes the issue around fault.

Examples:

“Who caused this?”
“Which person failed?”
“Which team is responsible?”

Responsibility matters, especially in real systems. But when blame comes too early, it can block structural understanding.

The question should not only be:

“Who did this?”

but also:

“What conditions made this likely?”

2.13.4 The False Choice Frame

This frame presents a situation as if only two options exist.

Examples:

“Should we choose quality or speed?”
“Should I stay or leave?”
“Should we build or buy?”
“Should we centralize or decentralize?”

Sometimes binary choice is necessary. But often the binary hides a wider decision space.

The better question may be:

“Under what conditions can we preserve enough quality while increasing speed?”

or:

“Which parts should be centralized, and which parts should remain local?”

2.13.5 The Totalizing Frame

This frame expands the problem until it becomes too large to handle.

Examples:

“My life is not working.”
“The company is broken.”
“Education is failing.”
“Society has lost direction.”

These statements may express real concern, but they do not yet support deliberate action.

A totalizing frame must be decomposed.

2.13.6 The Solution-First Frame

This frame smuggles a preferred solution into the problem statement.

Examples:

“How do we get everyone back to the office?”
“How do we make users use this feature?”
“How do I force myself to study more?”

These questions assume the solution before the problem has been examined.

A better formulation asks:

“What work requires physical presence?”
“What user need is this feature supposed to serve?”
“What prevents consistent study from happening?”

Good framing delays solution long enough to understand the problem.

2.14 The Grammar of Problem Framing

Problem framing can be treated as a grammar.

A basic problem frame can be written as:

In [context], [subject] experiences [undesired condition]
because / while / under [cause, constraint, or uncertainty],
which affects [stakes],
so the decision is [what must be clarified, changed, or chosen].

For example:

In the current learning environment, students struggle to explain ideas in English
because they lack reusable structures for organizing meaning,
which affects speaking fluency and confidence,
so the decision is how to design practice around structured explanation rather than isolated word memorization.

This frame is stronger than:

Students need better English.

Because it defines:

  • context

  • subject

  • condition

  • cause

  • stakes

  • decision direction

Another example:

In the product launch process, the team repeatedly misses deadlines
because major dependencies are discovered too late,
which creates rework and damages trust,
so the decision is how to surface dependencies earlier and sequence work more realistically.

This is stronger than:

The team is slow.

The grammar does not need to be used mechanically every time. But it reveals what a complete problem frame usually contains.

2.15 A Practical Method for Framing Problems

A useful problem-framing process can move through seven questions.

2.15.1 What is happening?

Begin with observation.

Do not interpret too quickly.

Ask:

  • What is occurring?

  • What signals are visible?

  • What changed?

  • What is repeated?

  • What is abnormal?

At this stage, the goal is not explanation. The goal is contact with reality.

2.15.2 Why is this considered a problem?

Identify the standard.

Ask:

  • What expectation is being violated?

  • What goal is being blocked?

  • What value is being threatened?

  • Who considers this a problem, and why?

This prevents vague dissatisfaction from pretending to be analysis.

2.15.3 What is the object of the problem?

Define the central object.

Ask:

  • Is this about a person?

  • A process?

  • A system?

  • A relationship?

  • A resource?

  • A constraint?

  • A decision?

  • A mismatch?

Changing the object changes the problem.

2.15.4 Where is the boundary?

Define scope.

Ask:

  • What is included?

  • What is excluded?

  • What level are we working at?

  • Is this local or systemic?

  • Is this short-term or long-term?

A frame without boundary cannot guide action.

2.15.5 What might be causing or sustaining it?

Move from symptom to structure.

Ask:

  • What conditions produce this?

  • What incentives maintain it?

  • What dependencies shape it?

  • What assumptions are involved?

  • What feedback loops exist?

Causality should be treated carefully. Early frames should allow revision.

2.15.6 What action paths does this frame imply?

Make the hidden direction visible.

Ask:

  • If we accept this frame, what will we try to do?

  • Who will be asked to change?

  • What resources will be used?

  • What alternatives will be ignored?

  • What risks does this frame create?

A frame should be judged partly by the quality of action it opens.

2.15.7 What alternative frames are possible?

Generate competing formulations.

Ask:

  • Can this be framed as a gap?

  • As a constraint?

  • As a trade-off?

  • As a coordination failure?

  • As a learning problem?

  • As a system design problem?

  • As an uncertainty problem?

The first frame is rarely the best frame.

2.16 Example: From Vague Problem to Deliberate Frame

Consider the statement:

“I cannot make progress.”

This is too vague for deliberate thinking.

It may be developed through several frames.

2.16.1 Frame 1: Motivation Problem

“I cannot make progress because I lack motivation.”

This directs action toward discipline, emotion, and willpower.

Possible actions:

  • create routines

  • increase accountability

  • use rewards

  • reduce friction

2.16.2 Frame 2: Goal Problem

“I cannot make progress because my goal is not clearly defined.”

This directs action toward clarification.

Possible actions:

  • define the desired outcome

  • set success criteria

  • distinguish primary and secondary goals

2.16.3 Frame 3: Decision Space Problem

“I cannot make progress because I see too many possible paths and lack criteria for choosing.”

This directs action toward option structuring.

Possible actions:

  • list alternatives

  • define criteria

  • compare trade-offs

  • choose a path for a fixed period

2.16.4 Frame 4: System Problem

“I cannot make progress because my environment continuously breaks concentration.”

This directs action toward context design.

Possible actions:

  • change workspace

  • block distractions

  • redesign schedule

  • reduce interruptions

2.16.5 Frame 5: Capability Problem

“I cannot make progress because the task requires skills I have not yet built.”

This directs action toward learning sequence.

Possible actions:

  • identify missing subskills

  • build practice loops

  • reduce task difficulty

  • get feedback

Each frame may be true in part. The task is not to choose one instantly, but to test which frame best explains the situation and opens useful action.

2.17 Framing and Responsibility

Problem framing also distributes responsibility.

This must be handled carefully.

A frame can place responsibility on:

  • the individual

  • the team

  • the system

  • leadership

  • environment

  • constraints

  • historical conditions

  • external forces

For example:

“The worker made a mistake.”

This assigns responsibility to the worker.

“The process allowed a single mistake to cause system failure.”

This assigns responsibility to process design.

“The organization did not create adequate review mechanisms.”

This assigns responsibility to governance.

The point is not to remove individual responsibility. The point is to locate responsibility accurately.

A mature frame can hold multiple layers at once:

The individual made an error,
but the system lacked safeguards,
and the organization had normalized a risky workflow.

This kind of frame supports better action because it avoids both extremes:

  • blaming only the individual

  • dissolving responsibility entirely into the system

Deliberate language must preserve responsibility while expanding understanding.

2.18 Framing and Time Horizon

Problems also change depending on time horizon.

A short-term frame asks:

“What must be fixed now?”

A medium-term frame asks:

“What pattern must be changed?”

A long-term frame asks:

“What structure must be redesigned?”

For example:

“We missed the deadline.”

Short-term frame:

“How do we recover this delivery?”

Medium-term frame:

“Why do we repeatedly underestimate project complexity?”

Long-term frame:

“How should our planning system change so that scope, dependency, and uncertainty are represented earlier?”

Each horizon has a different function.

Short-term framing supports response.
Medium-term framing supports correction.
Long-term framing supports redesign.

Bad deliberation often occurs when these horizons are confused.

A person may use long-term philosophical analysis when immediate action is required.
Or they may use short-term fixes when structural redesign is needed.

A good problem frame names the time horizon of concern.

2.19 Framing and Level of Analysis

A problem may exist at different levels.

For example, in a software product, a failure may be framed at the level of:

  • interface

  • feature

  • user workflow

  • product positioning

  • technical architecture

  • team process

  • business model

  • market timing

A surface-level frame might say:

“The button is confusing.”

A deeper frame may say:

“The workflow does not match the user’s mental model.”

An even deeper frame may say:

“The product assumes a user behavior that does not exist.”

Each level changes the action.

The art of framing is not always to go deeper. Sometimes the immediate level is enough. But deliberate thinking must know which level it is operating on.

The central question is:

At what level must the problem be framed in order for action to be effective?

If the frame is too shallow, action treats symptoms.
If the frame is too deep, action becomes abstract and slow.

A good frame has the right level for the decision at hand.

2.20 The Problem Frame as a Decision Gate

Problem framing is the gate through which the rest of deliberation passes.

Once a problem is framed, the next steps become possible:

  • goals can be clarified

  • criteria can be defined

  • options can be generated

  • alternatives can be compared

  • risks can be assessed

  • judgments can be made

  • plans can be formed

But if the frame is unstable, everything downstream becomes unstable.

A bad frame contaminates the entire decision process.

If the wrong problem is defined, the right solution may still fail.

This is why problem framing must be treated as a first-class act of deliberate language.

It is not a preliminary paragraph.
It is not a background section.
It is not merely context.

It is the construction of the object that judgment will operate on.

2.21 Problem Framing Template

A deliberate problem frame can be built using the following template:

The issue is not simply [surface symptom].
The deeper problem is [structured problem],
in the context of [situation],
where [subject/system] is trying to [goal],
but faces [constraint/conflict/uncertainty],
leading to [consequence/stakes].
Therefore, the central question is:
how to [decision/action direction] under [conditions].

Example:

The issue is not simply that users are inactive.
The deeper problem is that the product does not create a clear first success moment,
in the context of a crowded market where users abandon tools quickly,
where the product is trying to become part of a repeated workflow,
but faces unclear onboarding, weak feedback, and low perceived value,
leading to poor retention.
Therefore, the central question is:
how to redesign the early user experience so that users reach a meaningful result before attention is lost.

This template separates symptom, structure, context, goal, constraint, consequence, and decision direction.

It turns a vague concern into a deliberative object.

2.22 Minimal Form

In practice, a shorter version is often enough:

The problem is [structured condition],
not merely [surface symptom],
because [reason or sustaining structure],
which matters because [stakes].

Example:

The problem is a lack of reusable explanation structures,
not merely weak vocabulary,
because the learner cannot organize meaning quickly during speech,
which matters because real-time expression requires structure before fluency.

This minimal form is especially useful when beginning analysis.

It prevents premature solution and forces the problem to become explicit.

2.23 Chapter Summary

This chapter has argued that problem framing is the first act of deliberate language.

A problem is not simply given by reality. It is constructed through language by defining an object, boundary, standard, cause, agent, consequence, and direction.

Different formulations produce different paths of thought and action.

A weak frame may confuse symptoms with causes, compress complexity into labels, assign blame too early, assume false choices, or smuggle solutions into the problem statement.

A strong frame makes the issue thinkable, bounded, testable, and actionable.

The purpose of framing is not to make a problem sound better.

It is to make the problem structurally available to judgment.

Once the problem has been framed, deliberation can move to the next task:

clarifying goals, criteria, and stakes.
